# Proof

If you only have a few minutes to decide whether Aionis is real, start here.

This page collects the shortest defensible evidence path:

1. public benchmark disclosures
2. current docs verification status
3. the strongest applied runtime demo

That mix is deliberate. Good proof for a systems product should come from more than one layer. Public benchmark disclosures show what has already been published. Verification status shows what the current docs cycle actually re-ran against Lite. The runtime demo shows whether those surfaces matter in something closer to real usage than a bare HTTP example.

If any one of those layers stood alone, it would be too easy to overread it. Benchmarks without runtime shape can feel abstract. Runtime stories without verification status can feel hand-wavy. Endpoint verification without public evidence can feel too narrow. Put together, they form a much more defensible evaluation path.

## Fastest evidence path

```text
1. Read the public benchmark numbers
2. Inspect OpenClaw as the runtime proof
3. Check Verification Status for the current docs-cycle trust boundary
4. Only then open the API pages you care about
```

## Public benchmark proof

Current public material already supports these narrow claims:

1. continuation work used fewer tokens with Aionis in current disclosed A/B slices
2. cross-runtime handoff recovery improved on current published handoff slices
3. strict replay has already been shown as a zero-model-token path on the disclosed repository slice

Current disclosed numbers include:

1. `30.03%` lower input tokens on the published continuation slice
2. `77%` lower output tokens on the published continuation slice
3. `33.24%` lower total tokens on the published continuation slice
4. handoff recovery moving from `33.33%` to `100%`
5. strict replay at `0` model tokens on the disclosed repository slice

The word that matters here is narrow. These disclosures do not justify broad slogans like “Aionis always reduces tokens” or “Aionis always improves completion.” They justify something more serious: continuity discipline can produce measurable gains on disclosed slices, and Aionis already has public material supporting that claim.

That is enough to justify evaluation. It is also enough to separate Aionis from projects that are still selling only a conceptual architecture story.

## Runtime proof: OpenClaw

OpenClaw is the strongest applied Aionis proof today.

It shows Aionis shaping a real tool loop with:

1. completion uplift
2. churn reduction
3. replay routing
4. handoff fallback

Current public adapter evidence supports these narrow runtime claims:

1. live-task A/B reduced average executed steps from `7.33` to `3`
2. GLM-5 semi-live slices reduced average total tokens from `1893` to `865.33`
3. hard-stop and replay slices reduced average total tokens from `1659` to `1267`
4. current completion slices moved from `completed_rate = 0` to `completed_rate = 1`
5. repeated Google runtime-backed runs improved from `completed_rate = 0` to `0.8`

OpenClaw matters because it turns the product from static docs into observable runtime behavior. You can see context assembly before action, policy signals before tool choice, feedback after execution, and replay or handoff behavior when the loop should not simply continue forward. That is much closer to how a serious buyer or developer will judge the product than a one-off write/recall demo.

## Evidence ladder

Use the evidence in this order.

Start with the runtime story in [OpenClaw Adapter](/how/openclaw-adapter), because that is the easiest way to see Aionis acting on a real tool loop instead of a synthetic docs example. Then inspect [Benchmarks And SLO](/reference/benchmarks-and-slo) for the public benchmark layer, and use [Verification Status](/reference/verification-status) to see which parts of this site were re-run against Lite in the current cycle.

Only after that should you go to [API Reference](/api-reference/). The API pages are useful once you already believe the product is worth serious evaluation. They are not the fastest way to decide whether the product is real.
