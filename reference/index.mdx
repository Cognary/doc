# Reference

This is the deep layer for teams that need to decide whether Aionis is credible enough to integrate and operate.

This section should not be the first thing a new reader sees.

It becomes useful when someone asks harder questions:

1. which parts are actually verified
2. which endpoint surfaces are stable enough to wire
3. what identifiers must be logged
4. what the operational and governance boundary really is

That is a different mode of reading from the first three layers. Earlier pages are there to help you understand the product and get a first win. This section is here to remove ambiguity once the conversation becomes operational: what exactly is stable, what exactly was verified, and what exactly a team would need to log, monitor, and defend in production.

## Start here if you are integrating this week

Start with [Verification Status](/reference/verification-status), because it tells you which parts of this site were re-run and which are still code-backed. Then go to [Endpoint Reference](/reference/endpoints) for the surfaces your first integration will actually touch, use [API Reference](/api-reference/) when you need the broader family view, and only then go deeper into [Operations](/operations/) if you are already thinking about production rollout.

## How to use this section well

The biggest mistake in Aionis reference reading is treating every page as equally important on day one.

A better order is:

1. check [Verification Status](/reference/verification-status) so you know what is live-verified and what is only code-backed
2. read [API Contract](/reference/api-contract) so you understand the durable identifiers and route families
3. move into [Endpoint Reference](/reference/endpoints) only for the surfaces your first integration actually touches
4. read operations and governance only after the basic product path is already credible

That order keeps the reference useful instead of turning it into noise.

That is also why some pages here are narrower than people expect. A good reference page should answer one class of question clearly, not try to impersonate the whole product. The job of this section is precision, not persuasion.
